# AI 协作文档体系与工作流规范

## 一、需求阶段

### 沉淀文档：需求决策单

每个功能需求在动手前必须完成一张决策单，格式如下：

```markdown
## 需求决策单 — [功能名称]

### 问题陈述
一句话描述现状痛点（不是期望方案）

### 关键决策

| # | 决策点 | 选项 | 选定 | 原因 |
|---|--------|------|------|------|
| 1 | 交互形式 | A/B/C | B | ... |
| 2 | 数据存储 | A/B/C | C | ... |

### 不做的事
- 明确列出本次不实现的功能，防止 AI 自行扩展

### 影响范围
- 列出可能需要改动的现有文件/模块
```

**为什么用这个格式**：

- **问题陈述**迫使区分"痛点"和"方案"。AI 拿到痛点会主动提方案，拿到方案就只执行——后者容易做错方向
- **关键决策**是选择题而非开放式。AI 提供 2-3 个选项带 trade-off，人类 30 秒决策，比来回讨论 10 分钟高效
- **不做的事**是刹车线。AI 有"多做一点"的倾向，明确排除项防止范围蔓延
- **影响范围**让 AI 在动手前先扫描现有代码，避免遗漏需要同步修改的文件

**案例**：本项目"联系护理师"需求决策单：

> 问题陈述：点击"联系护理师"仅弹 toast，无实际交互
> 决策 1：交互形式 → 选定"文字+图片+快捷短语"，原因：实际场景需要发宠物照片
> 决策 2：数据存储 → 选定"localStorage 先行，预留实时接口"，原因：项目无后端
> 不做：图片/语音/快捷短语（YAGNI）、未读计数、WebSocket 实时同步
> 影响范围：owner/OrderDetail、caretaker/OrderDetail、App.jsx 路由

## 二、研发阶段

### 沉淀文档：任务拆解清单

需求决策单确认后，AI 产出任务拆解清单，格式如下：

```markdown
## 任务拆解 — [功能名称]

### 新增文件
| 文件 | 职责 | 依赖 | 预估复杂度 |
|------|------|------|-----------|
| src/data/chat.js | localStorage CRUD | 无 | 低 |
| src/pages/Chat.jsx | 聊天页面 | chat.js, useRole, useStore | 中 |

### 修改文件
| 文件 | 改动点 | 影响分析 |
|------|--------|----------|
| OrderDetail.jsx | handleContactCaretaker → navigate | 仅改一行，低风险 |
| App.jsx | 添加路由 | 新增路由不影响现有 |

### 任务依赖图
chat.js → Chat.jsx → 路由+导航接入
（chat.js 无依赖可先行，Chat.jsx 依赖 chat.js）

### 并行策略
- 可并行：chat.js（数据层）+ 路由注册（App.jsx 改动）
- 必须串行：Chat.jsx 依赖 chat.js 的接口，先确认接口再写组件
```

**并行处理原则**：

1. **按依赖关系分层**：无依赖的底层任务先行，上层任务等底层接口稳定后再动
2. **同层任务可并行**：数据层（chat.js）和路由注册（App.jsx）互不依赖，可以让 AI 同时处理
3. **接口先行**：底层模块的接口（函数签名、返回值、参数）先确认，上层模块才能正确调用。**接口确认不靠文档，靠跑测试**——写完 chat.js 立刻写 chat.test.js，测试通过即接口稳定

**案例**：本项目实际执行顺序：

```
第1步（并行）：chat.js + chat.test.js → 数据层接口稳定
第2步（并行）：App.jsx 路由 + owner OrderDetail 导航改动 → 接入层完成
第3步（串行）：Chat.jsx → 依赖 chat.js 接口 + 路由，最后写
第4步：caretaker OrderDetail 添加按钮 → 依赖路由存在
```

### 沉淀文档：接口变更日志

每次改动跨模块接口时，记录变更：

```markdown
## 接口变更 — [日期]

### 变更
addMessage(orderId, { sender, text }) → addMessage(orderId, { sender, text, image })

### 调用方同步状态
| 调用方 | 已同步 | 备注 |
|--------|--------|------|
| Chat.jsx | ✅ | 已传 image |
| [其他调用方] | ✅/❌ | ... |
```

**为什么需要这个**：addMessage 参数遗漏 image 导致图片发送后不显示——这类 bug 的根因是"定义方改了但调用方没同步"或反过来。接口变更日志强制 AI 在改接口时逐个检查调用方，防止静默丢数据。

## 三、测试阶段

### 核心原则：测试与研发并行，不是研发完再补测试

传统流程是"先写代码，再写测试"。AI 协作中应该反过来：**每个模块写完立刻写测试，测试通过后才算该模块完成**。

**并行策略**：

```
模块 A 代码 → 模块 A 测试 → ✅ 通过 → 模块 A 完成
                                    ↓
模块 B 代码 → 模块 B 测试 → ✅ 通过 → 模块 B 完成
```

不是：

```
模块 A 代码 → 模块 B 代码 → 模块 C 代码 → 全部测试 → ❌ 15 个失败 → 批量修
```

**原因**：AI 一次写大量代码后批量测试，失败原因会交叉污染（环境问题 + 接口问题 + 逻辑问题混在一起）。逐模块测试，每个失败原因独立定位，修复速度快 3-5 倍。

### 沉淀文档：测试覆盖清单

```markdown
## 测试覆盖清单 — [功能名称]

### 分层覆盖

| 层级 | 测试文件 | 测试数 | 覆盖要点 | 状态 |
|------|---------|--------|----------|------|
| 数据层 | chat.test.js | 8 | load/save/add/clear/isolate | ✅ |
| 页面层 | Chat.test.jsx | 8 | 空状态/发送/Enter/空消息/持久化/加载/图片 | ✅ |
| 接入层 | OrderDetail 改动验证 | — | 导航到 chat 路径 | ✅ 手动 |

### 环境基线
- setup.js: jest-dom + scrollIntoView mock
- jsdom 缺失 API 已统一 mock
```

### 如何通过测试校准实现完整性

测试不只是"代码能跑"，它是**需求覆盖度的校准工具**。每个测试对应一个需求点或边界条件，测试清单就是需求的可执行规格。

**校准方法**：

1. **需求→测试映射**：把需求决策单的每个决策点映射到具体测试
   - "支持图片发送" → `handlePhotoChange` 测试 + 图片消息渲染测试
   - "localStorage 存储" → `saveChat/loadChat` 测试 + 发送后 localStorage 验证
   - "空消息不发送" → 空输入 + disabled 按钮测试

2. **边界条件覆盖**：AI 写代码时容易遗漏边界，测试强制暴露
   - 空消息、空图片、同时有文字和图片、只有图片没有文字
   - localStorage 损坏数据、存储满、跨订单隔离

3. **接口契约验证**：测试是接口的活文档
   - `addMessage` 返回值结构、`loadChat` 空数据返回值、`saveChat` 异常处理
   - 如果接口改了但测试没同步，测试会立刻失败——比人工 review 更可靠

**案例**：addMessage 参数遗漏 image 的 bug，如果先写了 chat.test.js 再写 Chat.jsx，测试会在调用 addMessage 时暴露参数不匹配。但本项目是先写完所有代码再补测试，所以这个 bug 在测试阶段才被发现——**测试与研发并行能更早捕获此类问题**。

## 四、文档体系总览

| 阶段 | 文档 | 产出时机 | 作用 |
|------|------|---------|------|
| 需求 | 需求决策单 | AI 动手前 | 框定方向、排除范围、明确影响面 |
| 研发 | 任务拆解清单 | 决策单确认后 | 明确依赖、指导并行、防止遗漏 |
| 研发 | 接口变更日志 | 每次跨模块改动 | 强制同步调用方，防止静默丢数据 |
| 测试 | 测试覆盖清单 | 每个模块完成时 | 需求→测试映射，校准完整性 |
| 全程 | 设计文档 | 需求确认后 | 架构决策、数据模型、不做的事 |

### 文档流转规则

```
需求决策单 → 任务拆解清单 → [逐模块] 代码 + 测试 + 接��变更日志 → 测试覆盖清单
```

每份文档都是下一份的前置条件。**跳过任何一份，后续阶段的问题都会回溯到缺失的那份文档**：

- 跳过需求决策单 → 方向做错，返工
- 跳过任务拆解清单 → 并行顺序混乱，依赖冲突
- 跳过接口变更日志 → 跨模块数据丢失，静默 bug
- 跳过测试覆盖清单 → 需求点遗漏，功能"看起来完成但实际不完整"

---

## 五、一句话总结

**文档不是负担，是 AI 协作的控制杆——需求决策单定方向，任务拆解清单定节奏，接口变更日志防丢数据，测试覆盖清单校准完整性。每份文档都是下一阶段的前置条件，跳过任何一份都会回溯出问题。**
